## 内容

这是几条关于最近几周用 Claude 写代码时的一些零碎笔记。

### **Coding workflow（编程工作流程）**

鉴于大型语言模型（LLM）编码能力的最新提升，与许多人一样，我从 11 月大约 80% 手动 + 自动补全编码和 20% agent 编码，迅速提升到 12 月 80% 使用 agent 编码和 20% 手动编辑+修饰。也就是说，我现在大部分时间都在用英语编程，有点害羞地告诉 LLM 该写什么代码......用文字。这有点伤自尊，但能在软件上进行大规模“代码操作”的能力实在太有用了，尤其是一旦你适应它、配置它、学会使用它，并理解了它能做什么和不能做什么之后。这无疑是我这~二十年编程生涯中对基本编码工作流程最大的一次变化，而且发生在几周内。我预计，类似的转变将在更多工程师中发生，达到两位数百分比，而普通大众对此的认识还停留在个位数百分比。

### **IDEs/agent swarms/fallability（IDE/agent 蜂群/易失误）**

 对于“不再需要 IDE”和“agent 蜂群”的炒作，在我看来，目前都有些过头了。模型肯定还是会犯错，如果你有真正关心的代码，我建议你像鹰一样盯着它们，同时在一旁使用一个漂亮的大型 IDE。错误已经发生了很大变化——它们不再是简单的语法错误，而是一些稍显草率、仓促的初级开发者可能会犯的细微的概念错误。最常见的类别是，模型代表你做出错误的假设，然后就盲目配合，不加检查。他们也无法管理自己的困惑，不寻求澄清，不提出矛盾，不会发现不一致，不提出权衡，应该反驳时不反驳，而且它们仍然有点过于谄媚。在 Plan 模式（计划模式）下，会有所改善，但仍然需要一个轻量级的内联 Plan 模式。他们还喜欢把代码和 API 搞得太复杂，它们会过度抽象，不会自行清理死代码，等等。他们会在 1000 行代码中实现低效、臃肿、脆弱的结构，你得说“嗯，你不能直接做这个吗？”，他们会说“当然可以！”，然后立刻把代码缩减到 100 行。它们有时还会出现副作用，更改或删除它们不喜欢或不充分理解的注释和代码，即使这些内容与当前任务无关。尽管通过 `CLAUDE.md` 的说明进行了一些简单的尝试来修复这些问题，但所有这些情况仍然存在。尽管存在这些问题，它依然是一个巨大的进步，很难想象回到手工编码的时代。简而言之，每个人都有自己的开发流程，我现在的流程是在左侧的 ghostty 窗口/标签页中摆几个小型 CC 会话，右侧使用 IDE 查看代码并进行手动编辑。

### **Tenacity（坚韧）**

看着一个 agent 坚持不懈地工作，真是太有趣了。它们从不疲倦，从不气馁，只是不断前行，尝试那些人类早就放弃，只为迎接新的一天。看着它长时间的努力，最终在 30 分钟后才取得成功，这是一种“感受到 AGI（人工智能）力量”的时刻。你会意识到，耐力是工作的核心瓶颈，而有了大型语言模型（LLM）在手，耐力得到了大幅提升。

### **Speedups（提速）**

目前尚不清楚如何衡量大型语言模型（LLM）辅助的“提速”效果。当然，我觉得自己完成原定任务的速度确实快了，但主要影响是，我做的事情比原本多得多，因为：

1. 我能够编写各种以前不值得去写的东西。
2. 我能够处理以前因为知识和技能问题无法完成的代码。所以这当然是提速，但可能更像是一种扩展。

### **Leverage（筹码）**

大型语言模型（LLMs）在循环执行直至达到特定目标方面，表现非常出色，而这也正是“感受到 AGI（人工智能）力量”的魔力所在。不要告诉它该做什么，只需给出成功标准，然后静观其变。先让它编写测试，然后通过测试。将其与一个浏览器 MCP 一起放入循环中。先编写一个很可能正确的简单算法，然后要求它在保持正确性的同时进行优化。将你的方法从命令式改为声明式，以使 agent 循环更长时间并获得优势。

### **Fun（有趣）**

我没想到，有了 agent 编程后，感觉更有趣了，因为很多填空式的繁琐工作被去除了，剩下的就是创意部分。我也感觉不再那么受阻/卡住（这不好玩），而且我有了更多的勇气，因为几乎总是有办法与之携手合作，取得一些积极的进展。当然，我也从其他人那里看到了相反的情绪，大型语言模型（LLM）编程会让工程师们分裂成两类，一类主要喜欢编码 (coding)，另一类主要喜欢构建 (building)。

### **Atrophy（萎缩）**

我已经注意到，自己手写代码的能力正在慢慢退化。生成（写代码）和辨别（读取代码）是大脑的不同能力。很大程度上是因为编程中涉及的所有小细节，主要是语法细节，即使你在编写代码时敢到困难，你仍然可以很好的审查(review)代码。

### **Slopacolypse（废料末日?）**

我预计 2026 年将是 GitHub、Substack、arxiv、X/Instagram 以及所有数字媒体遭遇“滑坡灾难”的一年。我们还会看到更多的 AI 炒作生产力的戏码（这可能吗？），而实际上，真正的进步却寥寥无几。

注："Slop-acolypse" 是一个合成词，由 "slop"（废料/剩菜）和 "apocalypse"（末日）组合而成，通常用来戏谑地描述因废料堆积或食物浪费引发的灾难性场景。

### **Questions（问题）**

我脑海中浮现的几个问题：

- “10X 工程师”的情况如何——即平均工程师与最高效率工程师之间的生产力比率——会发生什么？这个比率可能会大幅增长。
- 在大型语言模型（LLMs）的加持下，多面手的表现是否越来越优于专家？与宏观的大战略相比，大型语言模型在填空（微观）方面表现出色得多。
- 未来使用大型语言模型（LLM）进行编程会是什么感觉？是像玩《星际争霸》？《星露谷物语 Factorio》？还是像演奏音乐？
- 数字知识工作在多大程度上制约了社会的发展？

### **TLDR（总结）**

这给我们带来了什么？LLM agent 能力（尤其是 Claude 和 Codex）在 2025 年 12 月左右跨越了某种一致性门槛，并在软件工程及相关领域引发了阶段性的转变。智能部分突然感觉远远领先于其他部分——集成（工具、知识）、新组织工作流程的必要性、流程、更广泛的传播。随着行业对新能力的吸收，2026 年将会是充满活力的一年。
